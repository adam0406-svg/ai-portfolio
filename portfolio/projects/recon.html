The Problem

Sales reps walk into meetings underprepared. Not because they don't care — because real research takes too long.

A proper pre-meeting company briefing requires cross-referencing LinkedIn for key people, scanning news sources for recent developments, reading annual reports or press releases, synthesising pain points, and translating it all into a relevant pitch angle. Done properly, this takes two to three hours per prospect.

In practice, it doesn't get done. Sales reps skim. Consultants arrive with generic decks. Customer success teams go in blind. The result is meetings that feel underprepared, discovery questions that should have been answered before walking in the door, and trust that takes longer to build because the other side can tell you haven't done your homework.

"We spend more time doing research we'll forget by the time the meeting starts than we spend in the actual meeting." — the pattern this tool is designed to eliminate.

The opportunity is clear: if this research could be automated without losing quality, it would both save significant time and raise the floor for every client interaction. No more underprepared meetings. No more reading a company's homepage in the taxi.

The Solution

Four AI agents. One company name. A complete briefing in 90 seconds.

RECON is a Sales Meeting Preparation tool built on a 4-agent CrewAI pipeline. You type a company name (and optionally upload context about yourself or your angle), and the system runs four sequential AI agents that each handle a distinct aspect of research and synthesis.

The Researcher agent queries live web data via Tavily — news, recent announcements, key people, products, and publicly visible pain points. The Writer agent takes those raw findings and structures them into a JSON briefing format. The Strategist turns that briefing into a tailored pitch angle, differentiators, and a recommended engagement approach. The Devil's Advocate stress-tests the whole thing — surfacing likely objections, risk flags, and intel gaps you should be aware of before you walk in.

The output is a 9-slide presentation deck rendered directly in the browser: company overview, recent intelligence, who you're meeting, pain points, your strategy, objections and counters, and talking points. Each slide is scroll-friendly and the deck works on any screen. The time saved counter runs in real-time while the agents work — a live comparison against the 150 minutes this would have taken manually.

Technical Approach

Why CrewAI, and how the pipeline is structured.

This project was specifically chosen to introduce AI framework usage and a true agentic tool-use loop — two capabilities absent from earlier projects that relied on direct API calls. CrewAI manages the multi-agent orchestration: agent roles, sequential execution, and structured output passing between agents.

Stack & Architecture
Backend
FastAPI (Python 3.12) — two endpoints: POST /api/research runs the full pipeline; POST /api/extract-text handles optional file uploads (.pdf, .docx, .txt)
Framework
CrewAI orchestrates 4 agents in sequence: Researcher → Writer → Strategist → Devil's Advocate. Each agent has defined roles, goals, and backstory; CrewAI handles the execution loop and output passing.
AI Model
Claude (Anthropic) via litellm format for CrewAI compatibility. Each agent runs the same model but with different system prompts and structured output schemas defined in task definitions.
Search
Tavily API provides live web search results to the Researcher agent — recent news, company pages, executive profiles. This is the agentic tool-use component: the model calls search and decides what to investigate.
Frontend
Single-file index.html — no build step, no framework. Three views: search input, live loading (with real-time time-saved counter), and the 9-slide deck. Keyboard navigation on the deck (← →).
Parsing
Custom _strip_json and _safe_parse utilities handle LLM output that may include prose or code fences around JSON. If one agent fails, it returns {} rather than crashing the whole run — resilience by design.
Lessons & Pitfalls

What went wrong, and what it taught me.

!

JSON parsing from LLMs is messier than expected

Language models frequently wrap JSON output in prose or code fences, even when instructed not to. The initial implementation crashed on every non-clean response. The fix was a dedicated regex-based extractor that strips surrounding text — but the real lesson was: never trust that an LLM will follow output format instructions perfectly. Always parse defensively.

!

Agent slide layouts broke when I added position: relative to the wrong elements

The deck uses position: absolute on slide elements to enable the horizontal slide transition. Adding position: relative to any child class overrode this and caused slides to stack in document flow, being clipped by the deck overflow. This class of CSS conflict is subtle and time-consuming — a good reminder to understand the positioning model before layering overrides.

i

Tavily returns fewer results for lesser-known companies

For large, well-documented companies (Novo Nordisk, Maersk, Stripe), research quality is excellent. For smaller or private companies, Tavily finds less, and the Researcher agent produces thinner output. This is a structural limitation of web-scraping approaches — public footprint determines research quality. The system handles it gracefully, but it's worth communicating as a constraint when pitching this use case.

i

The time-saved counter is disproportionately persuasive

Adding a real-time display of "time saved vs. manual" — watching 2 hours accumulate in 90 seconds — makes the value proposition visceral in a way that a static stat cannot. This was initially added as a UI flourish but turned out to be the single most memorable element of the demo. Show people the value, don't just tell them.

What's Next

Roadmap for a production version.

The current version is a single-user local tool. A production version would need several additional layers to work in an organisational context.

  • CRM integration — trigger RECON automatically when a prospect is added to HubSpot or Salesforce, and write the briefing back to the record
  • Custom research templates — allow organisations to define their own briefing structure and focus areas (e.g. financial services teams want different intel than SaaS sales teams)
  • Persistent company profiles — cache research for accounts the team visits repeatedly, with staleness detection that re-runs when content is older than N days
  • Team sharing — share briefings with a link, comment on slides, flag items for follow-up
  • Regenerate per slide — re-run individual agent sections without rerunning the full pipeline
  • PDF export — download the deck as a formatted PDF to share or print for travel
